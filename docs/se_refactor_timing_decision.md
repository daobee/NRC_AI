## 1. 重构内容概要

### 1.1 是什么

**SkillTiming (SE) 重构** = 给技能效果添加显式的"触发时机"阶段，类似特性的 `AbilityEffect` 架构。

**当前状态**:
- 技能效果是平铺 `List[EffectTag]`（无时序语义）
- 条件判断散落在 handler 内部（if 分支）
- 应对效果用特殊容器类型（COUNTER_ATTACK/STATUS/DEFENSE）

**重构后**:
- 技能效果包装为 `SkillEffect(timing, effects, filter)`
- 时机由结构表达：PRE_USE, ON_USE, ON_HIT, ON_COUNTER, IF, POST_USE
- 条件从 handler 迁移到 filter 字典
- 应对改为 ON_COUNTER timing + category filter

### 1.2 影响范围

| 组件 | 修改量 | 破坏性 | 恢复难度 |
|-----|-------|-------|--------|
| effect_models.py | +2 类 (SkillTiming, SkillEffect) | 无 | N/A |
| effect_engine.py | +100 行 + 改 200 行 | 中 | 中 |
| effect_data.py | 改 50+ 技能配置 | 高 | 低 |
| skill_effects_generated.py | 重新生成 455 技能 | 高 | 低 |
| battle.py | 删 100+ 行硬编码 | 高 | 中 |
| 生成器 scripts/generate_skill_effects.py | +200 行 | 低 | 低 |

**总体**: 代码改动 ~500-800 行，涉及 6 个文件

## 2. 与 P0 的关系

### 2.1 是否阻塞 P0

**P0 目标**: 伤害公式精度 ≤ 5% 偏差

**结论**: ❌ **不阻塞**

原因：
- SE 重构只改变效果触发的"时机结构"
- 不改变伤害计算公式本身
- 不改变 DamageCalculator 逻辑
- 所有现有测试仍然有效

### 2.2 是否加速 P0

**结论**: ❌ **不加速**

原因：
- P0 主要工作：采集游戏数据 + 调整参数
- SE 重构是代码质量改进，不涉及数据
- 实际上可能减速（需要改动 50+ 技能配置）

## 4. 时机选择

### 4.1 选项 A: 立即启动（P0 前）

**优点**:
- SE 架构更清晰，便于后续特性配置
- P0 期间如发现设计缺陷可及时调整
- 完成后的代码更易维护

**缺点**:
- P0 验收时间推后 1-2 周
- 增加不必要的风险（可能影响伤害公式测试）
- 如出现问题需回滚，浪费时间

**时间成本**: 3-5 天（步骤 1-5）

### 4.2 选项 B: P0 后启动（并行 P1）

**优点**:
- P0 数据校准不受影响
- 如 P0 暴露新需求，SE 设计可调整
- 更低风险的实施

**缺点**:
- P1 特性实现需等待 SE 完成
- 代码维护成本（同时有旧新两套格式）

**时间成本**: 同样 3-5 天，但更安全

### 4.3 **建议方案：延后启动（选项 B）**

**推理**:

1. **P0 是关键路径** — 伤害公式精度决定整个模拟器的可信度
2. **SE 是非关键路径** — 代码质量改进，不影响功能
3. **最小化风险** — 不在 P0 期间引入大量改动
4. **最大化并行度** — 特性补全（P1）可在等待 SE 期间继续

**决策**:
- ✅ P0 (周 1-2): 伤害公式 + 精灵数值 + 基础特性
- ✅ P1a (周 2-3): 补全 TOP 20 特性（用现有 effect_data 格式）
- ✅ P1b (周 3-4): 启动 SE 重构 + 迁移现有技能
- ✅ P2 (周 4+): 继续特性补全 + 应对系统完善

## 5. 重构 9 步的可执行性

### 5.1 每步风险评估

| Step | 任务 | 风险 | 验证方式 |
|-----|-----|------|--------|
| 1 | effect_models.py (新增) | 低 | 导入检查 |
| 2 | effect_engine.py (兼容) | 中 | 现有测试通过 |
| 3 | 手工技能迁移 (50个) | 低 | 技能配置检查 |
| 4 | 生成器改造 | 中 | 生成结果对比 |
| 5 | 重新生成 (455技能) | 低 | 文件行数检查 |
| 6 | 废弃 COUNTER_* | 高 | 应对系统测试 |
| 7 | 清理 handler | 中 | 逻辑测试 |
| 8 | 清理 battle.py | 高 | 完整战斗测试 |
| 9 | 测试补充 | 低 | 测试覆盖率 |

### 5.2 最小化风险方案

建议分两个 PR:

**PR-1 (兼容阶段)**:
- Step 1-3: 新增结构 + 手工技能迁移
- 保持向后兼容（旧平铺格式仍可用）
- 测试：新格式能正常工作

**PR-2 (清理阶段)**:
- Step 4-9: 废弃旧格式 + 清理硬编码
- 只在 PR-1 验证后进行
- 测试：所有现有测试通过

**这样做的好处**:
- PR-1 如有问题易于回滚（只删不改）
- PR-2 改动更大但在安全基础上进行
- 任何 bug 可追溯到具体 PR
